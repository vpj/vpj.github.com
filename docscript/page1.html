Markdown like syntax for Edward Tufte style documents

We started working on a documentation engine to create documents such as printed user manuals, getting started guides, online help, training handouts and internal documents at Foresptin.

Initially we were working with editors such as Microsoft Word. It was alright when we had a couple of documents but it wasn't easy to maintain formats and manage them as larger documents came in. Then we started looking at alternatives like LaTeX and Markdown.

Both these options would make managing the documentations much easier. The documents could be version controlled with git, which lets us do a number things: version control, branching, collaboration, etc. Although LeTeX gives a lot of flexibility, it was a little too complicated, especially for non-technical writers.

Markdown on the other hand was much simpler but didn't support some of the key features we wanted. Markdown doesn't work well when printing. It breaks pages random places. The other problem was that Markdown didn't support sidenotes.

R Markdown suported formatting inspired by Tufte. This was the best already available option for us. We honestly didn't have very good reasons not to use it and develop our own.

However there were a few advantages of developing our own tool:

  • Give us total control

    We would be able to modify the tool to perfectly fit our needs

  • Mardown doesn't have a structure

    For instance, in the following document,

    #Heading1
Intro
#Heading2
Paragraph
#Heading3
Paragraph
Conclusion

    it is not clear whether Conclusion belongs to Heading1 or Heading3. This again gives some trouble when paginating.

  • R Markdown seemed to be use verbose syntaxes for some of the commonly needed functions

DocScript Project

Docscript is available on Github. It is not fully baked yet but you can give it a try.

Here are some of the sample documents we've created:

  • Benford's Law Test
  • Forestpin Dashboard
  • Correlation Test

Usage

The online compiler is available at http://vpj.github.io/docscript/.

The command line interface requires nodejs and coffeescript. You need to get the git submodules with git submodule init and git submodule update after cloning docscript.

./docscript.coffee -i input_file -o output_file

This will create the output html file inside build directory. The CLI is still in early stages.

Design

Docscript uses indentation to specify hierarchy of content.

Example

###Heading1

 Introduction

 * Point one

  Description about point one

 ####Subtopic

  Subtopic content

  More subtopic content

 This belongs to Heading1

Althought indentation doesn't help much in standard rendering except in a a few cases (e.g. lists) it's has a number of of other uses. It lets us programmetically set page breaks when printing (not implemented yet). Other advantage is we can use code folding when editing; which come handy when working with large documents.

Another key feature of docscript is sidenotes. You can have notes as well as images in sidenotes.

We've changed some syntaxes of Markdown; for instance, << and >> are used for links instead of [](), because we felt it was a little more intuitive (resemblence with HTML tags).

Docscript introduces HTML blocks marked by <<<, where you can add any HTML content.

Reference

Headings

Headings have the same syntax as Markdown. # for level 1 headings ## for level 2 headings and so on. The content that belongs to the heading should indented.

#Heading

 Indented content

Paragraphs

Again similar to Markdown. Paragraphs are separated by blank lines.

Paragraph1
More of paragraph1

List

Ordered lists begin with - while unordered lists begin with * . Since lists can be structured with indentations, it's easy to have lists within lists.

- Analyses

 - Timeseries analysis
- Visualizations

Text can be made italics with --.

Superscript and Subscript

Superscript are wrapped inside ^^ ^^ and Subscripts are wrapped inside __ __.

* 2^^2^^ = 4
* CO__2__

Inline code

Inline code is identified by two backticks (`).

Click ``apply`` to save changes.

Links

Links are wrapped inside << >>. The link text can be specified within brackets.

* <<http://www.twitter.com/vpj(My Twitter Account)>>

Visualizing numerical tables 2

Here's a number of alternatives to displaying table of numbers. We have evaluated them in terms of,

  1. Reading numbers
  2. Finding the largest and smallest
  3. Getting an idea about the distribution

If there is only one column of numbers, the table can be sorted by that. But there is a bit of a problem when there are multiple columns and you want to see the relationship between two columns. In which case you will have to sort by one of the columns and understand the distribution of the other column relative to that.

Listing numbers

The simplest and often the best way is to simply list them. Right aligned numbers help quick scanning and comparison.

If the reader wants to get an idea about the distribution or range of numbers in a non-sorted column she will have to go through the each of the numbers.

Bars

Adding a bar, like in a horizontal bar chart, helps you clearly identify the largest value and understand the distribution without having to read through the numbers.

The bars take up a lot of space - at least another column. Displaying text over bars reduces the space, but it makes it lot harder to read the numbers.

Underline

A thin line underneath the numbers instead of the bars can give a visual indication of the values, without taking much space.

Right aligning the bars eliminated minimizes clutter. A subtle line color and/or thinner lines will help reduce distraction even more.

The distribution of numbers can be identified even when viewing from a distance or with a blurry vision.

The scale of the lines could be linear or logarithmic depending on the distribution of the numbers. For better understanding, a small axis could be placed at the column headings.

Benford's Law Tests

Benfordâ€™s Law states that for financial data, the frequencies of digits should conform to given expected frequencies. This theory is based on the fact that large quantities occur less frequently than small quantities.

Benfordâ€™s Law tests direct the investigator to find errors, fraud and biases of systematic or repetitive nature. For example, a fraud that has been carried out over a period of time with a similar pattern, a systematic error which has been repeating frequently or a bias such as upper limit on payments to certain vendors.

Forestpin runs last two digits test and first two digits test, comparing the first two digit and last two digit frequencies with expected frequencies, to find any abnormal patterns in invoice values. Moreover, Forestpin also has last two digit summation test and first two digit summation test, which compare the summation of invoices instead of frequencies. This gives a different perspective to finding patterns and can be used with the respective frequency tests, to find errors and fraud.

First Two Digits Test & First Two Digits Summation Test

These tests are based on Benfordâ€™s Law. First Digits Test and First Two Digits Summation Test can find errors, irregularities and frauds, which are repetitive or systematic.

Example Findings

Employees making multiple payments for values right below a certain threshold

e.g.: Approval is required for payments above $50 and employees have submitted large number of payments for $49. This signals fraud or irregularities.

Breaking down large unauthorized payments into chunks of payments

e.g.: A payment of $200,000 is paid with 200+, $900-$999 payments, because payments above $1000 need authorization.

If bills are required for payments over $20 there may be a lot of payments just below $20 without bills.

How It Works

Consider the following dataset (amounts):

$119, $110, $4345, $23, $98, $23000
